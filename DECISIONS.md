# Architecture Decision Records (ADRs)

Short, dated, reversible. Each records *what* we decided and *why*, grounded in the June 2026 research.

## ADR-001 — Read JSON endpoints, don't scrape a background browser
**Decision:** Default fetch path is a native `URLSession` call to the provider's JSON endpoint, authenticated with a credential read from the Keychain. The "open Chrome in background and reload" idea is demoted to a last-resort fallback adapter.
**Why:** A proven reference (`ttar-p/claude-usage-widget`) reads the Claude Code OAuth token from Keychain and calls `api.anthropic.com/api/oauth/usage` directly. This is ~6 MB vs hundreds of MB of bundled Chromium, survives UI redesigns, and avoids JIT entitlements / nested-binary notarization hell.
**Trade-off:** Endpoints are undocumented/unversioned and can break; we mitigate with last-good caching and clear auth-expiry handling.

## ADR-002 — WidgetKit cannot do 60s; menu bar app is the flagship
**Decision:** The true-60s experience is the **menu bar app** (and Übersicht). The Notification Center WidgetKit widget is shipped as a *glanceable, ~15-min* surface with honest copy.
**Why:** Apple gives widgets a daily budget of ~40–70 reloads (effective ~15–60 min); minimum entry spacing ~5 min; even host-app `reloadAllTimelines()` is throttled and deferred. Debugger has no limit, which misleads. This is a hard platform constraint, not a bug we can engineer around.
**Trade-off:** We must set user expectations; we market the widget as "glanceable," not "live."

## ADR-003 — All-Swift shared core (no bundled Node/Chromium by default)
**Decision:** `FetcherCore` is a Swift package consumed directly by the menu bar app and widget; the Übersicht `.jsx` calls a tiny Swift CLI (`houdini`) or curls directly. No Node/Playwright in the default build.
**Why:** Bundling Chromium roughly doubles engineering effort *just for code signing* (bottom-up signing, JIT entitlements `allow-jit` / `allow-unsigned-executable-memory`, library-validation disable). All-Swift keeps hardened-runtime signing trivial. Only adopt a bundled headless browser if a target provider truly has no readable endpoint.

## ADR-004 — Claude first, OpenAI/Console secondary, ChatGPT Plus experimental
**Decision:** Build the Claude Pro/Max adapter first (cleanest signal). Anthropic Console + OpenAI Platform admin-API adapters second (zero ToS risk, documented). ChatGPT Plus quota is experimental/best-effort.
**Why:** Claude exposes `utilization_pct` directly behind the user's progress bars. OpenAI has no clean consumer quota number; only "limit reached" signals. Don't promise a gauge we can't reliably fill.

## ADR-005 — Credentials in Keychain, never on a server
**Decision:** All tokens/cookies stored in macOS Keychain; no Houdini backend ever receives them. Embedded WebView login for cookie-based providers; offer to reuse Claude Code's OAuth token when present.
**Why:** The app touches logins; trust is the core adoption barrier. This also becomes the landing-site privacy story.

## ADR-006 — Distribution: ad-hoc-signed app via install.sh (curl|bash) is the shipping path; notarized DMG + Homebrew cask (+ Sparkle) remain a future option
**Revised 2026-07-01** (in place; supersedes the original "notarized-DMG-for-MVP" framing below). Original decision was provisional and tagged "revisit after signing setup"; reality has settled, so this ADR now records what production actually ships.
**Decision:** Production ships a single **ad-hoc-signed `Houdini.app`** distributed via **`install.sh` (`curl | bash`)**, pinned to the current release tag. This is the **intended** distribution channel today, not a stopgap — it is the concrete expression of ADR-010's single-version rule (one advertised version, one install path, installer always re-fetched from the current tag). `install.sh` keeps its advertised guarantees: it **verifies SHA-256** against `SHASUMS256.txt` before install, installs **without `sudo`**, avoids a **Gatekeeper prompt** (ad-hoc-signed; quarantine handled so there is no multi-step System Settings ordeal), and **offers** a launch-at-login item rather than forcing it. A **signed + notarized DMG** ($99 Apple Developer Program), a **Homebrew cask** secondary channel, and **Sparkle** auto-update remain a documented **future** option, to be adopted if/when broader reach or one-click DMG install is wanted — not a blocker for the current release.
**Why:** The one-liner installer gives non-technical users a clean, one-command install today with integrity + least-privilege guarantees, and no paid Apple program is required to ship it. Keeping exactly one advertised version + one install path is the smallest correct distribution surface (ADR-010), and re-fetching `install.sh` from the current tag means the pinned one-liner never points at a dead artifact. The notarized-DMG path (and Homebrew, which also requires notarization and mainly reaches technical users) buys a nicer double-click install and auto-update, but at the cost of the $99 program and more signing machinery; it is worth revisiting later, but is not needed for the current single-version release.
**Consequence / trade-off:** `curl | bash` asks users to trust a piped install script; we counter that with the SHA-256 verification, no-`sudo` install, and an open, auditable `install.sh` (see `RELEASE.md`). This ADR does not add a Houdini server or move any credential off-device (ADR-005 / ADR-011 unchanged), and it does not add any site-side install UI beyond the single one-liner the site already advertises (consistent with ADR-010/011).
**Future decision (deferred, not blocking):** if we later want one-click DMG install and Sparkle auto-update, decide then whether to pay the $99 Apple Developer Program and stand up the notarize + staple + cask pipeline.

## ADR-007 — Pluggable provider adapters
**Decision:** Every provider implements one `UsageProvider` protocol with capability flags (`usagePct`, `resetTimer`, `dollarBalance`). The registry + UI adapt to whatever a provider can supply.
**Why:** Providers differ wildly (Claude has %, OpenAI Platform has $, ChatGPT Plus has almost nothing). A capability-driven model lets us add providers without touching the UI core.

## ADR-008 — Mobile is a native iOS app (cookie auth), not a PWA
**Decision:** The iPhone version is a **native SwiftUI app** (+ WidgetKit Home/Lock Screen widgets) that signs into claude.ai in an in-app `WKWebView`, captures the `sessionKey` cookie, stores it in the **iOS Keychain**, and calls the same `claude.ai` JSON endpoints via native `URLSession`. It reuses `FetcherCore`'s **cookie** path verbatim (`ClaudeCookieProvider` + `ClaudeUsageParser`); there is no Claude Code OAuth token on a phone, so the cookie path — the Mac app's *fallback* — becomes the iOS *primary*. FetcherCore was made iOS-compilable by guarding its only macOS-only dependency (`Foundation.Process`, in `CredentialStore.cliReadGenericPassword` and `ClaudeOAuthProvider.detectedClientVersion`) behind `#if os(macOS)`; the macOS build is unchanged (see `apps/ios/PLAN.md` §5).
**Why a native app, not a PWA:** A browser can't do this honestly. (1) **CORS** — a page on `houdini.salomao.org` calling `claude.ai/api/.../usage` is cross-origin and claude.ai doesn't allow it; native `URLSession` is not a browser and isn't subject to CORS. (2) **httpOnly** — `sessionKey` is an httpOnly cookie, unreadable from `document.cookie`; only a privileged native cookie store (`WKHTTPCookieStore`) can read it. The only way to make a PWA work is a **server proxy holding the user's cookie**, which would break the core privacy promise (ADR-005). So native is the *only* honest path.
**Trade-off / consequences:** (a) **Refresh** — iOS widgets inherit the same Apple budget as ADR-002 (~15–30 min, *not* 60s); the app refreshes on open/foreground; copy stays "updated X min ago". We promise 60s nowhere on iOS. (b) **Distribution** — unlike the Mac app there is **no free install-by-link**: shipping to others requires the **Apple Developer Program ($99/yr) + Xcode**, via TestFlight then the App Store (extends ADR-006; iOS removes macOS's un-paid escape hatches). (c) **Android** is a *future*, separate Kotlin app using the same cookie approach — not built now. None of the build/ship steps are possible on this CommandLineTools-only machine, so today's work is scaffold + an iOS-ready core only.

## ADR-009 — Rebrand "Tally" → "Houdini"
**Decision:** Rename the product, repo, app, CLI, and all user-facing surfaces from **Tally** to **Houdini** (`v0.2.0`). The menu-bar SPM target is `HoudiniApp` — case-distinct from the `houdini` CLI target, because SwiftPM keys each target's build tree off `<TargetName>.build/` and on case-insensitive APFS `Houdini.build/` and `houdini.build/` are the *same* folder, so equal-but-for-case names collide and the app's objects never get emitted. The shipped bundle binary is still `Houdini`. Bundle id → `org.salomao.houdini`; Keychain service → `Houdini-claude-session`. No credential migration — the user base is ~0.
**Why:** (1) **Collision + saturation** — "Tally" already names a shipping AI-usage tool (`ai-tally.app`) and is a heavily-used generic app word; it offered no defensible identity. (2) **Positioning** — the product is broadening from "Claude usage" to a **multi-provider AI usage + cost** platform; a distinctive, ownable name with a clear metaphor (the escapist who **reveals what's hidden** → "see your usage and spend, revealed") carries that story far better than a literal "tally". (3) **Identity** — "Houdini" unlocks a premium, characterful brand (top-hat mark, stage spotlight, gold "reveal" spark) that a utilitarian name did not.
**Trade-off / consequences (accepted):** "Houdini" is a crowded search term — **SideFX Houdini** (3D software) and **CSS Houdini** (web APIs) dominate organic results, so discovery leans on the `houdini.salomao.org` domain, the precise product descriptor in titles/OG, and direct/word-of-mouth rather than ranking for the bare word. Judged worth it for a memorable, on-metaphor brand over a generic, already-taken one. `tally.salomao.org` 301-redirects to the new domain so no links break.

## ADR-010 — Single-version production hygiene (no stale releases, no "coming soon")
**Decision:** Production carries exactly **one** advertised version at a time. Every production release: (1) **bumps** the version, (2) **publishes** the new release + verified installer, and (3) **removes** every obsolete GitHub release, git tag, and installer artifact so they can't be installed by mistake. The site, `README`, and the install one-liner reference **only the current version**. Production never ships a "coming soon" / "Soon" / "next milestone" placeholder; a surface is either real and shown, or absent. When a feature ships, the UI is **adapted** to absorb it (re-frame the relevant section) rather than having a new loose block stacked on. The procedure is the checklist in `RELEASE.md`. See ADR-009 for the rename that motivated the first cleanup (the obsolete `v0.1.0`/`v0.1.1` "Tally" releases carried broken one-liners and were deleted under this rule).
**Why:** Multiple live releases mean multiple install paths, and old pinned one-liners keep "working" while pointing at dead/renamed artifacts — every stale tag is a support liability and a trust dent. "Coming soon" copy in production reads as vaporware and ages badly; honest products show what exists today. One version + one install path is the smallest correct surface.
**Trade-off:** Deleting old tags breaks any deep-link to a retired release (accepted — the user base is ~0 and `install.sh` is always re-fetched from the current tag). Contributors must follow `RELEASE.md` precisely; an interrupted release can briefly leave the repo between versions, so the bump+publish+prune steps are scripted to run as one unit.

## ADR-011 — Multi-provider direction; provider API keys never touch the frontend
**Decision:** Houdini is a **multi-provider** AI-usage + cost tool. The user-visible product is a single app with a **provider switcher in the app's own Settings** (not on the website). **Claude (Pro/Max)** is the primary, live provider (Keychain OAuth token or a one-time claude.ai session — see ADR-005 / `PROVIDERS.md`). **OpenAI Platform** is the planned **second** provider, connected with an **organization admin key**. That admin key — and any provider API/admin key, ever — is entered **only in the native app's Settings and stored only in the macOS Keychain**. It is **never** placed in the website, in any frontend/JS bundle, in environment variables shipped to the browser, in `config.ts`, or anywhere in the repo / version control. The website shows **no per-provider key UI and no visible OpenAI placeholder**; capability is expressed as one honest line ("Works with Claude today — built to track every AI you pay for").
**Why:** (1) **Security** — an API/admin key in frontend or repo code is a credential leak by construction; it would be world-readable in the bundle or git history. The only place a secret belongs is the OS Keychain on the user's own machine, read by native code (extends ADR-005: no Houdini server, no secret in transit to us). (2) **Honesty** — a dead provider grid with "Soon" badges advertises capability that isn't there (ADR-010); one honest capability line is truthful and ages well. (3) **Architecture** — the capability-driven adapter model (ADR-007) means a new provider is added in the core + Settings, never by reshaping the marketing site.
**Trade-off:** OpenAI's admin-API surfaces org-wide cost/usage, not a clean per-user gauge (ADR-004), so its UX is "$ this period + tokens", not a utilization %. The switcher and the OpenAI adapter are **not built in this round** — this ADR fixes the direction and the hard "keys never on the frontend/site/repo" rule so later implementation can't drift.

## ADR-012 — Claude subscription auth (OAuth token / claude.ai cookie) is under Terms-of-Service review; freeze expansion
**Status:** Open / under review — logged 2026-07-01. Interim holds in effect; durable direction TBD by the owner.
**Context:** Houdini's live Claude provider (v0.4.0) reads the user's Claude Code **OAuth token** (Keychain `"Claude Code-credentials"` / `~/.claude/.credentials.json`) or a **claude.ai session cookie**, then polls Anthropic's usage endpoints (`api.anthropic.com/api/oauth/usage` / `claude.ai/api/organizations/{org}/usage`) about every 60s. In early 2026 Anthropic clarified and began enforcing that (1) Consumer Terms §3.7 forbid automated/script access to the Services except via an Anthropic **API key** or where explicitly permitted, and (2) OAuth tokens obtained through Free/Pro/Max accounts may not be used "in any other product, tool, or service" — OAuth is intended only for Claude Code and claude.ai. Enforcement is active (account bans; OpenCode removed Pro/Max support citing Anthropic legal requests). Both of Houdini's current Claude auth paths therefore appear to fall within the prohibited use, independent of the (unwired) token refresh. There is no ToS-sanctioned API for an individual's Pro/Max **subscription** limits (5h/weekly) — those are exposed only via Claude Code's `/status` and the claude.ai UI; the only sanctioned programmatic usage data is the **Admin Usage & Cost API** (`/v1/organizations/usage_report/...`, Admin API key), which reports **API/Console** consumption, not subscription limits.
**Decision (interim):**
1. Do **not** wire the OAuth token-refresh default (`refresher` stays `nil`; see P1 slice (a)).
2. **Freeze** expansion of the subscription-OAuth/cookie path — no first-run PKCE "Connect" (candidate c), no cookie-flow hardening (candidate b) that deepens reliance on it — pending a direction decision.
3. Treat the shipping Claude subscription-usage integration as **under ToS review**, with the user-account-ban risk called out.
4. Choose a direction before P2, from: **(i)** migrate to Anthropic **API-key / Admin Usage & Cost API** (ToS-clean, but shows API usage/cost, **not** Pro/Max subscription limits); **(ii)** seek **explicit permission** from Anthropic for a local, read-only subscription-usage viewer (the only path that preserves today's headline feature); **(iii)** accept the risk (not recommended); **(iv)** de-emphasize subscription reads and lead with API-key providers (OpenAI / Anthropic Console).
**Why:** The plain text of the Consumer Terms + the Claude Code legal-and-compliance clause prohibit third-party use of subscription OAuth; Anthropic is banning accounts and cites "unusual traffic without the Claude Code harness's telemetry" — which describes Houdini's ~60s polling. Shipping this risks **Houdini's own users' Anthropic accounts** and contradicts Houdini's "safe by design / trust-first" posture. Fair caveat: enforcement targets inference/token-arbitrage harnesses, not a local read-only usage viewer using one's own credential — but the terms carve out no such exception. This ADR is a risk/decision record, not legal advice.
**Consequence / trade-off:** The sanctioned path cannot reproduce Houdini's current headline feature (Pro/Max limits at a glance) — it reports API usage/cost instead — so (i)/(iv) change what the product leads with, and preserving the headline feature realistically requires (ii). Once a direction is chosen, `CONTEXT.md`, `README.md`, `PROVIDERS.md`, and the site's provider/trust copy (and the "no account" claim) must be revised to match. Related: ADR-005 (cookie/Keychain posture).
